## Critical blocking findings

### 1. T+1 execution is still applied in the wrong chronological order

Inside the signal-date loop, the simulator retrieves D+1 open prices, updates cash and holdings as if D+1 fills have occurred, and then marks the portfolio using the current loop date D close. This means future D+1 holdings are present in the D equity row. A logged `fill_date > order_date` does not repair this accounting-time error.

Required fix:

- queue orders on D;
- mark D close using pre-fill holdings;
- execute queued orders only when the event loop reaches D+1;
- then mark D+1 close using post-fill holdings.

### 2. Execution and tradability rules remain incomplete

- D+1 rows with `open = NaN` are not explicitly rejected because the code checks dictionary membership, not price validity.
- suspension status is not loaded or checked;
- D+1 limit/suspension/tradability is not evaluated;
- ST status is inferred from `list_status` containing `ST` without a committed field contract proving that semantic;
- sell-side limit-down and execution restrictions are not enforced.

### 3. Cost and cash accounting remain incomplete

- sell-side commission is not deducted, although the packet says commission applies to trades;
- the buy condition allows cost up to `per_stock * 1.1`, which may create negative cash;
- retained positions are not resized to equal target weights at rebalance;
- cost reconciliation artifacts report aggregate commission/stamp values, not the required row-level pre-value minus all costs equals post-cash plus holdings identity;
- slippage is not included in the committed reconciliation CSV.

### 4. Preregistered strategy fidelity is still not proven

- H3 does not implement an explicit breadth guard; its test remains an empty `pass`.
- all signals may produce a 20% bucket, but the simulator truncates every strategy to `top_n=10`.
- suspension/listing/delist/liquidity-floor and sector-neutral requirements remain unimplemented.
- the fidelity CSV is generated by declaring every active hypothesis `MATCH`, rather than deriving fidelity from behavioral checks.

### 5. Soft and hard allocators are not implemented as claimed

The code still constructs:

- `SOFT_PROBABILITY_WEIGHTED` by taking an arithmetic mean of eight independent strategy equity arrays;
- `HARD_CONFIRMED_STATE` by taking an arithmetic mean of four independent strategy equity arrays.

The soft allocator does not consume daily GMM probabilities. The hard allocator does not switch by the daily confirmed state. Neither maintains a shared allocation ledger or allocator-level traded-notional costs.

Additional evidence problems:

- `allocator_equity_curves.parquet` contains summary metric rows rather than daily equity curves;
- target and realized weight files contain only ten rows of fixed 0.5/0.3/0.2 aggregate weights;
- the switch log contains one hard-coded transition;
- `DEFENSIVE_FALLBACK` is merely H4, not a low-confidence fallback allocator.

Therefore `real_allocators=true` is rejected.

### 6. Behavioral-test claims are unsupported

`tests/test_c2r4_behavioral.py` still includes weak or empty checks:

- H3 breadth test is `pass`;
- unavailable-open test asserts `len(rejects) >= 0`, which is always true;
- H8 test asserts `len(signals) >= 0`, which is always true;
- cost test checks only non-negative aggregate values, not reconciliation;
- no-test-leak test checks only date ordering;
- packet-commit test checks repository HEAD, not the packet's `code_commit`;
- cash/holdings test checks only that portfolio value is positive.

The 22-row behavioral test matrix is generated by marking every requirement `VERIFIED`; it is not generated from pytest results. No GitHub Actions workflow or committed pytest transcript exists for the R4 commit.

### 7. Test-selection audit is tautological

The code defines `test_metrics` from rows whose split is `test`, then checks whether any test row exists outside that same list. This must return false and does not prove that test metrics were unused by model, allocator, packet, or label selection.

### 8. GMM remediation is only partial

The cluster-to-state mapping is now train-only, which is accepted as progress. However:

- `n_components` is chosen from the number of states observed in the full sample, allowing validation/test state occurrence to affect model architecture;
- scaler/model parameters and hashes are not preserved;
- daily out-of-sample probabilities are not committed as an auditable artifact;
- five GMM clusters map to only two semantic states, limiting regime coverage;
- the claimed soft allocator does not use these probabilities.

### 9. Strategy Intake Packets are not system-intake valid

Both R4 packets use:

`code_commit = fae383317c35f807c728078ccddda019eba523bc`

which is the progress-report commit, not the audited R4 implementation commit. They also:

- claim `pit_status=PIT_VERIFIED` while the DB callback and circ_mv PIT audit are pending;
- use a hard-coded snapshot label rather than an immutable database callback;
- omit export hashes and exact generator command;
- use generic formulas rather than exact implemented signal formulas;
- omit diagnostic test metrics;
- do not prove packet-to-artifact integrity.

The packet-schema file merely says `static_valid=true` and `regime_valid=true`; it does not validate the required fields or semantic references.

### 10. Exit gates are hard-coded rather than evidence-derived

The R4 script assigns several gates directly to `True`, including:

- `real_allocators`;
- `t1_execution`;
- `cost_traded_notional`;
- `packet_schema_valid`;
- `tests_behavioral`;
- `no_db_leak`.

These values conflict with the code and evidence above. Only family-count and drawdown gates are honestly false.
